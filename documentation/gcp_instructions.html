Every major provider now requires you to register an "application" in their specific developer ecosystem.

To Google, you are a GCP user. To Microsoft, you are an Azure user. To Yahoo, you are a YDN (Yahoo Developer Network) user.

Comparison: Where to go for each provider

The logic remains the same across all of them: you are creating a "Client ID" and "Client Secret" to identify your APP instance.

Provider Platform Name Setup Difficulty Key Terms to Look For
Gmail Google Cloud (GCP) Moderate "OAuth Client ID", "Desktop App"
Outlook / Hotmail Azure Portal (Entra ID) High "App Registrations", "Mobile & Desktop Applications"
Yahoo / AOL Yahoo Developer Network Low "Create an App", "Desktop App"
Fastmail (Internal Dashboard) Very Low "App Passwords" or "OAuth Clients"

Microsoft (Outlook/Hotmail) through Azure

Microsoft’s process is notoriously more corporate than Google’s. You don't technically need a paid "Azure subscription", but you do need to use the Azure Portal (now called Microsoft Entra ID).

  • The Difference: In Azure, you must specifically select "Accounts in any organizational directory and personal Microsoft accounts" during setup. If you choose "Single Tenant," your app will only work for your own business domain, not your @outlook.com address.
  • Redirect URI: Unlike Google, which handles "Desktop" automatically, Microsoft often asks for a specific "Redirect URI." For NeoMutt, you usually use "http://localhost"

Yahoo / AOL (Yahoo Developer Network)

Yahoo doesn't have a full "Cloud Platform" like Google or Microsoft, but they have a dedicated Developer Network.

  • The Difference: Yahoo is currently very restrictive. They have actually paused new "Mail" scope registrations for many individual users recently.
  • The Workaround: Many Linux users find that Yahoo still supports "App Passwords" more reliably than OAuth2 for third-party CLI clients. If OAuth2 fails, you generate a 16-character password in your Yahoo account security settings and use that instead.

Why does this keep happening?

Accessing email is no longer just "User + Password." It is now "User + Application + Provider." By forcing you into Azure or GCP, these companies ensure that:

  1. They know the app (NeoMutt).
  2. They know the person responsible for the app (You).
  3. They can revoke access: They can kill your specific "Client ID" without affecting your actual password.
App Type Typical Use Case Where the "Secret" Lives Why you aren't using it
Desktop App Mutt, NeoMutt, CLI tools, scripts running on your PC. In your local config file on your hard drive. This is the one you need. It assumes the app can open a local browser to let you log in.
Web Application Websites like Airbnb or Spotify ("Login with Google"). On a secure remote server (hidden from the user). Requires a "Redirect URI" (a web URL) to send the login code back to. NeoMutt isn't a website.
Android / iOS Mobile apps installed from the Play Store or App Store. None (Uses the App's unique ID/Fingerprint instead). These don't use "Client Secrets" because you can't hide a secret inside a phone app's code.
TV / Limited Input YouTube on a Smart TV, or a device with no keyboard. Hardcoded in the device firmware. Uses a "Device Code" (e.g., "Go to a URL and type XJ49"). Too slow for a daily mail client.
Service Account Automated "Bot" scripts that run in the background (no human). A private JSON key file. Gmail ignores these for personal accounts. You can't use a Service Account to read personal @gmail.com mail for "security reasons."

Creating an app in GCP (around 7 minutes):

Phase 1: Create the Project

  1. Go to: https://console.cloud.google.com/welcome
  2. Top Left: Click the Project Dropdown (it usually says "Select a project" or shows your last project name).
  3. Top Right of Pop-up: Click "New Project".
  4. Field "Project Name": Enter My-NeoMutt-Project.
  5. Click: The blue "Create" button at the bottom.
    • Wait about 10 seconds for the notification bell in the top right to show a green checkmark.
  6. Crucial Step: Click the project dropdown again and select your new project to make sure you are working inside it.

Phase 2: Enable the Gmail API

This tells Google that this "App" is allowed to ask for email access.

  1. Top Search Bar: Type "Gmail API"
  2. Results: Click on the one that simply says "Gmail API" (under Marketplace).
  3. Center Screen: Click the big blue "Enable" button.
    • The screen will refresh and take you to the "API/Service Details" page.

Phase 3: The OAuth Consent Screen (Identity)

This is where you tell Google who you are.

  1. Sidebar (Left): Click "APIs & Services" > "OAuth consent screen"
  2. User Type: Select "External".
  3. Click: "Create"
  4. Section "App information":
    • App name: Type NeoMutt
    • User support email: Select your email from the dropdown.
  5. Section "Developer contact info":
    • Email addresses: Type your own email address again.
  6. Click: "Save and Continue" at the bottom.
  7. Scopes Page: Click "Save and Continue" (we don't need to add them here manually; the mail client will request them).
  8. Test Users Page:
    • Click "+ ADD USERS".
    • Type your email address (abc@gmail.com).
    • Click "Add"
    • Note: If you don't do this, you will get a "403 Access Denied" error later.
  9. Click: "Save and Continue" one last time.

Phase 4: Generate Your Credentials (The Keys)

This is the information NeoMutt actually needs to talk to Google.

  1. Sidebar (Left): Click "Credentials"
  2. Top Center: Click "+ Create Credentials" and select "OAuth client ID"
  3. Application Type Dropdown: Select "Desktop app".
  4. Name: You can leave it as "Desktop client 1".
  5. Click:"Create".
  6. The "Success" Pop-up: A window will appear with your Client ID and Client Secret.
    • Action: Copy both of these into a safe text file immediately.

What you have now:

You now have the Client ID and Client Secret. This is the "App Identity" that NeoMutt will present to Google.

When you run NeoMutt's OAuth script, it will ask for these two strings. Once provided, it will open a browser window asking you to "Allow" your app to access your mail. Because you added yourself as a Test User, it will let you through.
